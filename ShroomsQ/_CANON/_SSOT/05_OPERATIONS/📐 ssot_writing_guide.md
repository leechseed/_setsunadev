## Purpose

This document establishes the authoritative standard for Single Source of Truth (SSOT) documentation. It ensures all frameworks remain canonical, precise, and functionally optimized for the Obsidian Properties UI through rigid linguistic and structural constraints.

---

## Linguistic Authority Standards

To maintain doctrinal authority, all SSOT documentation must adhere to the following constraints:

- **No Contractions**: All words must appear in complete form (e.g., "do not," "it is").
- **No Hedging**: Qualifiers such as "basically," "perhaps," or "maybe" are prohibited.
- **Present Tense**: Methodology is described as an active, perpetual reality.
- **Technical Precision**: The first instance of a technical term must be **Bolded** and defined immediately.
- **Active Voice**: Subjects must perform specified actions without ambiguity.

---

## Structural Requirements

Every SSOT document must follow the standardized H1-H2 hierarchy:

1.  **YAML Boundary**: Must begin on Line 1 with `---`.
2.  **Document Title**: H1 prefixed with 📐 and `ssot_`.
3.  **Table of Contents**: Linked hierarchical navigation.
4.  **Purpose Statement**: Single paragraph defining document scope.
5.  **Core Methodology**: The logical rules and mechanics.
6.  **Implementation**: Step-by-step application procedures.
7.  **Examples**: Minimum of two hypothetical implementations.
8.  **Version History**: Tabular record of revisions.

---

## The Flat-Syntax YAML Standard

To guarantee Obsidian Property rendering, all metadata must use **Flow Style** for lists:

- **Lists**: `[Value 1, Value 2]`
- **Wikilinks**: `"[[Document_Name]]"` (must be double-quoted).
- **Key Spacing**: `key: value` (mandatory space after colon).

## Purpose

This document defines the operational standards and workflow protocols for the **Goon Room**, the primary environment for narrative synthesis, rapid prototyping, and systemic state analysis.

---

## Operational Environment

The Goon Room functions as a high-intensity synthesis chamber. It is designed for the intersection of raw story data and rigorous methodology.

- **Input**: Raw narrative concepts, character seeds, and systemic variables.
- **Process**: Refinement through SSOT frameworks and Marine Corps doctrinal scrubbing.
- **Output**: Canonical documentation and Authoritative project files.

---

## Workflow Procedures

Practitioners in the Goon Room follow a cyclical three-phase process:

1.  **Ingestion**: Retrieval of legacy data and current project requirements.
2.  **Synthesis**: Application of the 15-category Narrative Astrology and 6-Movement Lifecycle.
3.  **Standardization**: Conversion of synthesized data into Obsidian-compliant Markdown files.

---

## Data Integrity Standards

Every piece of data generated within the Goon Room must pass a **Canonicity Audit**:

- **Consistency**: The data does not contradict existing SSOT methodologies.
- **Traceability**: Every authoritative file must reference the specific SSOT it employs.
- **Formatting**: Strict adherence to the `📐 ssot_` or `🎯 auth_` naming conventions.

---

## Implementation

1.  **Initialize Environment**: Open Obsidian and ensure the Mission Control dashboard is active.
2.  **Select Movement**: Identify which of the 6 movements is currently being rendered.
3.  **Execute Framework**: Apply relevant SSOT logic to the narrative problem.
4.  **Finalize Output**: Use the External Save Protocol (VS Code) if Properties fail to render.
