# Process Groups Overview

## The Big Picture: How Process Groups Relate

The five Process Groups are not a one-and-done sequence. They **overlap and interact** throughout the project. Here's how they relate:

```
                    ┌─────────────────────────────────────────────────────┐
                    │                   PROJECT                           │
                    │                                                     │
  Business         │  ┌──────────┐                        ┌──────────┐  │   Final
  Documents ──────►│  │INITIATING│                        │ CLOSING  │  │──► Product/
  (Business Case,  │  │          │                        │          │  │   Result/
   Benefits Plan)  │  └────┬─────┘                        └────▲─────┘  │   Lessons
                    │       │                                   │        │   Learned
                    │       ▼                                   │        │
                    │  ┌─────────────────────────────────────────┐       │
                    │  │              PLANNING                   │       │
                    │  │                                         │       │
                    │  │  (24 processes — the largest group)     │       │
                    │  │                                         │       │
                    │  └──────────┬──────────────────────────────┘       │
                    │             │            ▲                         │
                    │             ▼            │  (change requests       │
                    │  ┌──────────────────┐    │   feed back into       │
                    │  │    EXECUTING     │    │   planning)            │
                    │  │                  │    │                         │
                    │  │ (10 processes)   │────┘                         │
                    │  └──────────┬───────┘                              │
                    │             │                                      │
                    │             ▼                                      │
                    │  ┌──────────────────────────────┐                  │
                    │  │  MONITORING & CONTROLLING    │                  │
                    │  │                              │◄──── Continuous  │
                    │  │  (12 processes — runs        │      throughout  │
                    │  │   across everything)         │──────────────────┘
                    │  └──────────────────────────────┘
                    └─────────────────────────────────────────────────────┘
```

**Key interactions:**

- **Initiating** feeds into **Planning** (charter and stakeholder register inform all planning)
- **Planning** feeds into **Executing** (the plan guides the work)
- **Executing** generates **change requests** that feed back into **Planning**
- **Monitoring & Controlling** runs continuously, comparing actuals against the plan, and triggers changes
- **Closing** formally wraps up, archiving information and releasing resources

## Process Groups Are NOT Phases

This is worth repeating from Step 1 because it's a common point of confusion:

|Process Groups|Project Phases|
|---|---|
|Logical groupings of _processes_|Logical groupings of _project activities_|
|All 5 can occur within a single phase|Each phase has its own deliverable(s)|
|Defined by the PMBOK|Defined by the organization/project|
|Always the same 5|Can be any number/name|

**In a multi-phase project,** the Process Groups repeat within each phase.

# Initiating Process Group (2 Processes)

## Initiating Overview & Purpose

The Initiating Process Group defines a new project (or new phase) and obtains authorization to start. Its purpose is to:

- Align stakeholder expectations with the project's purpose
- Define initial scope and commit initial financial resources
- Identify stakeholders who will interact with and influence the project
- Appoint the project manager (if not already assigned)
- Show organizational commitment to the project

**Key benefits:**

- Only projects aligned with strategic objectives get authorized
- Business case, benefits, and stakeholders are considered from the very start

**In multi-phase projects:** Initiating processes are revisited at the start of each phase to verify the business case is still valid, review stakeholder expectations, and confirm the project remains focused on its business need.

> ⚠️ **The project charter is NOT a contract** — no money or consideration is exchanged. But it _formally authorizes_ the project and gives the PM authority to use organizational resources.

## 4.1 Develop Project Charter

|Aspect|Detail|
|---|---|
|**Purpose**|Formally authorizes the project; links project to strategic objectives|
|**Key Benefit**|Creates a formal record showing organizational commitment|
|**Performed**|Once, or at predefined points|
|**Key Inputs**|Business documents (business case, benefits management plan), agreements|
|**Key Output**|Project charter, assumption log|
|**Who Creates It**|Sponsor or PM in collaboration with the initiating entity|

**Critical points:**

- The PM is identified and assigned as early as feasible — preferably while the charter is being developed
- The PM does NOT update the business documents (they're not project documents), but may make recommendations
- Projects are initiated by entities **external to the project** (sponsor, PMO, portfolio governing body)

## 13.1 Identify Stakeholders

|Aspect|Detail|
|---|---|
|**Purpose**|Identify who can impact or be impacted by the project|
|**Key Benefit**|Enables appropriate focus for engagement of each stakeholder/group|
|**Performed**|Periodically throughout the project (starts in Initiating, continues throughout)|
|**Key Inputs**|Project charter, business documents, PM plan components, project documents|
|**Key Output**|Stakeholder register, change requests|

**Critical points:**

- Should commence **as soon as possible** after the charter is approved and PM is assigned
- This is an iterative process — stakeholders change throughout the project
- Updates may trigger changes to the communications, risk, and stakeholder engagement plans

# Planning Process Group (24 Processes)

## Planning Overview & Purpose

The Planning Process Group establishes the total scope, refines objectives, and develops the course of action to achieve them. The processes here produce the **project management plan** and **project documents** that guide execution.

**Key concepts:**

- Planning is **iterative** — as more information emerges, planning may be revisited. This is called **progressive elaboration**.
- Change requests from Executing or M&C may trigger re-planning
- When initial planning is complete, the approved PM plan becomes the **baseline** against which performance is measured
- The project team should seek **stakeholder input and involvement** during planning

## Integration Planning (1 Process)

### 4.2 Develop Project Management Plan

|Aspect|Detail|
|---|---|
|**Purpose**|Defines, prepares, and consolidates ALL subsidiary plans into one integrated plan|
|**Performed**|Once or at predefined points|
|**Key Output**|Project management plan (the master document)|

The PM plan is the **single source of truth** for how the project will be executed, monitored, and controlled. It includes all subsidiary management plans (scope, schedule, cost, quality, resource, communications, risk, procurement, stakeholder, change, configuration), baselines (scope, schedule, cost, performance measurement), and descriptions (life cycle, development approach).

## Scope Planning (4 Processes)

|#|Process|Key Benefit|Performed|
|---|---|---|---|
|5.1|**Plan Scope Management**|Provides guidance on how scope will be managed|Once|
|5.2|**Collect Requirements**|Provides the basis for defining scope|Once|
|5.3|**Define Scope**|Describes product/service/result boundaries and acceptance criteria|Once|
|5.4|**Create WBS**|Provides a framework of what has to be delivered|Once|

**Flow:** Plan Scope → Collect Requirements → Define Scope → Create WBS

The WBS is the **deliverable-oriented hierarchical decomposition** of all project work. Once created, the WBS + WBS dictionary + project scope statement = **scope baseline**.

---

## Schedule Planning (5 Processes)

|#|Process|Key Benefit|Performed|
|---|---|---|---|
|6.1|**Plan Schedule Management**|Provides guidance for schedule management|Once|
|6.2|**Define Activities**|Decomposes work packages into schedule activities|Throughout|
|6.3|**Sequence Activities**|Defines logical sequence for greatest efficiency|Throughout|
|6.4|**Estimate Activity Durations**|Provides the time each activity will take|Throughout|
|6.5|**Develop Schedule**|Generates the schedule model with planned dates|Throughout|

**Flow:** Plan Schedule → Define Activities → Sequence Activities → Estimate Durations → Develop Schedule

> 💡 **Notice:** Processes 6.2–6.5 are performed "throughout the project" (not just once), because schedules evolve as the project progresses — especially with rolling wave planning.

## Cost Planning (3 Processes)

|#|Process|Key Benefit|Performed|
|---|---|---|---|
|7.1|**Plan Cost Management**|Provides guidance for cost management|Once|
|7.2|**Estimate Costs**|Determines monetary resources required|Periodically|
|7.3|**Determine Budget**|Establishes authorized cost baseline|Once|

**Flow:** Plan Cost → Estimate Costs → Determine Budget

The cost baseline = the time-phased authorized budget. Once approved, it can only change through formal change control.

Developed early in the project but reviewed and revised regularly. The plan considers each stakeholder's information needs, format preferences, and timing requirements.

## Risk Planning (5 Processes)

|#|Process|Key Benefit|Performed|
|---|---|---|---|
|11.1|**Plan Risk Management**|Ensures risk management is proportionate to project risks and importance|Once|
|11.2|**Identify Risks**|Documents individual risks and sources of overall project risk|Throughout|
|11.3|**Perform Qualitative Risk Analysis**|Focuses efforts on high-priority risks|Throughout|
|11.4|**Perform Quantitative Risk Analysis**|Quantifies overall project risk exposure|Throughout|
|11.5|**Plan Risk Responses**|Identifies appropriate ways to address risks; allocates resources|Throughout|

> 💡 **Risk has more Planning processes (5) than any other Knowledge Area.** This reflects that risk management is fundamentally a planning discipline — though risk processes are performed iteratively throughout the project.

**Qualitative vs. Quantitative:**

- **Qualitative** = prioritize risks by assessing probability and impact (every project should do this)
- **Quantitative** = numerically analyze the combined effect on overall objectives (not every project — typically for large, complex, or high-risk projects)

# Executing Process Group (10 Processes)

## Executing Overview & Purpose

The Executing Process Group completes the work defined in the PM plan. This is where:

- **The majority of the budget, resources, and time are spent**
- Resources are coordinated and stakeholder engagement is managed
- Activities are performed in accordance with the PM plan
- **Change requests are generated** (which may trigger re-planning)

## The Executing Process Flow

```
From Planning ──►  Direct and Manage Project Work (4.3)
                   (the "engine" — leads all work, implements approved changes)
                          │
          ┌───────────────┼───────────────────────────────┐
          │               │               │               │
          ▼               ▼               ▼               ▼
    Manage Project   Manage Quality   Acquire        Manage
    Knowledge (4.4)  (8.2)            Resources      Communications
                                      (9.3)          (10.2)
                                         │
                                    ┌────┤
                                    ▼    ▼
                               Develop  Manage
                               Team     Team
                               (9.4)    (9.5)
          │               │               │               │
          ▼               ▼               ▼               ▼
    Implement Risk   Conduct          Manage Stakeholder
    Responses (11.6) Procurements     Engagement (13.3)
                     (12.2)
                          │
                          ▼
                   Change Requests ──► Back to Planning
```

## All 10 Executing Processes

|#|Process|KA|Key Benefit|Performed|
|---|---|---|---|---|
|4.3|**Direct and Manage Project Work**|Integration|Overall management of project work and deliverables|Throughout|
|4.4|**Manage Project Knowledge**|Integration|Leverages prior knowledge; makes new knowledge available|Throughout|
|8.2|**Manage Quality**|Quality|Increases probability of meeting quality objectives|Throughout|
|9.3|**Acquire Resources**|Resource|Guides resource selection and assignment|Periodically|
|9.4|**Develop Team**|Resource|Improved teamwork, skills, motivation, performance|Throughout|
|9.5|**Manage Team**|Resource|Influences behavior, manages conflict, resolves issues|Throughout|
|10.2|**Manage Communications**|Communications|Efficient information flow between team and stakeholders|Throughout|
|11.6|**Implement Risk Responses**|Risk|Ensures risk responses are executed as planned|Throughout|
|12.2|**Conduct Procurements**|Procurement|Selects a qualified seller and implements legal agreement|Periodically|
|13.3|**Manage Stakeholder Engagement**|Stakeholder|Increases support and minimizes resistance|Throughout|

> 💡 **Mnemonic — "DDMAADM-ICM" is terrible, so try this instead:** **"Directors Know Quality. Acquire, Develop, Manage teams. Communicate. Implement risks. Conduct procurements. Manage stakeholders."** Or just remember: Integration has 2, Quality has 1, Resource has 3, Comms has 1, Risk has 1, Procurement has 1, Stakeholder has 1.

**Key observations:**

- **Direct and Manage Project Work** is the central "engine" of executing — it leads and performs all work and implements approved changes
- **Manage Project Knowledge** is a PMBOK 6th Edition addition — reflects the growing importance of knowledge management
- **Resource has 3 executing processes** (Acquire, Develop, Manage) — more than any other KA in this group
- Scope, Schedule, and Cost have **no Executing processes** — their "doing" is inherent in the work itself

# Monitoring & Controlling Process Group (12 Processes)

## M&C Overview & Purpose

The M&C Process Group tracks, reviews, and regulates project progress and performance. It identifies variances from the plan and initiates corrective actions. This group:

- **Runs continuously throughout the project** (not just periodically)
- Compares actual performance against planned performance
- Analyzes variances and assesses trends
- Evaluates alternatives and recommends corrective/preventive action
- Influences factors that could circumvent change control

**M&C also involves:**

- Evaluating change requests and deciding on responses
- Recommending corrective or preventive action before problems occur
- Monitoring activities against the PM plan and baselines
- Ensuring only **approved changes** are implemented

> 💡 **Monitoring** = collecting data, producing performance measures, reporting information **Controlling** = comparing actual vs. planned, analyzing variances, recommending corrective action

## The M&C Process Flow

```
From Executing ──►  Monitor and Control Project Work (5.1)
                    (the "dashboard" — tracks overall progress)
                           │
                           ▼
                    Perform Integrated Change Control (5.2)
                    (the "gatekeeper" — reviews ALL change requests)
                           │
          ┌────────────────┼────────────────────────────────┐
          │                │                │               │
          ▼                ▼                ▼               ▼
    Validate Scope   Control Schedule  Control Costs   Control Quality
    (5.3)            (5.5)             (5.6)            (5.7)
    Control Scope
    (5.4)
          │                │                │               │
          ▼                ▼                ▼               ▼
    Control Resources Monitor Comms    Monitor Risks   Control
    (5.8)             (5.9)            (5.10)          Procurements
                                                       (5.11)
                                            │
                                            ▼
                                       Monitor Stakeholder
                                       Engagement (5.12)
                                            │
                                            ▼
                                     Change Requests ──► Back to Planning
```

## All 12 M&C Processes

|#|Process|KA|Key Benefit|Performed|
|---|---|---|---|---|
|4.5|**Monitor and Control Project Work**|Integration|Stakeholders understand current state; visibility into future status|Throughout|
|4.6|**Perform Integrated Change Control**|Integration|Documented changes considered in integrated manner|Throughout|
|5.5|**Validate Scope**|Scope|Objectivity in acceptance; increases probability of final acceptance|Periodically|
|5.6|**Control Scope**|Scope|Scope baseline is maintained|Throughout|
|6.6|**Control Schedule**|Schedule|Schedule baseline is maintained|Throughout|
|7.4|**Control Costs**|Cost|Cost baseline is maintained|Throughout|
|8.3|**Control Quality**|Quality|Verifies deliverables meet requirements for final acceptance|Throughout|
|9.6|**Control Resources**|Resource|Ensures physical resources are available as planned|Throughout|
|10.3|**Monitor Communications**|Communications|Optimal information flow as defined in comms plan|Throughout|
|11.7|**Monitor Risks**|Risk|Decisions based on current risk information|Throughout|

**Key observations:**

- **Every Knowledge Area has at least one M&C process** (except Integration which has 2)
- **Perform Integrated Change Control** is the gatekeeper — ALL change requests from ANY process flow through here
- **Validate Scope** is the formal acceptance process — stakeholders sign off on completed deliverables
- **Control Quality → Validate Scope** is a critical flow: quality checks produce _verified_ deliverables, which are then _accepted_ through scope validation

### The Three Baselines and Their Control Processes

|Baseline|What It Measures|Control Process|Together They Form|
|---|---|---|---|
|**Scope baseline**|WBS + WBS dictionary + scope statement|Control Scope (5.6)||
|**Schedule baseline**|Approved schedule model|Control Schedule (6.6)|**Performance Measurement Baseline**|
|**Cost baseline**|Time-phased authorized budget|Control Costs (7.4)||

> 💡 **The performance measurement baseline** = scope baseline + schedule baseline + cost baseline. This is what Earned Value Management (EVM) measures against.

# Closing Process Group (1 Process)

## Closing Overview & Purpose

The Closing Process Group formally completes or closes the project, phase, or contract. It verifies that all processes across all Process Groups are complete and formally establishes closure.

**This group also handles early closure** — aborted or cancelled projects still need to go through closing activities.

## 6.1 Close Project or Phase

|Aspect|Detail|
|---|---|
|**Purpose**|Finalize all activities for the project, phase, or contract|
|**Key Benefits**|Information is archived, planned work is completed, resources are released|
|**Performed**|Once or at predefined points|
|**Key Inputs**|All PM plan components, all relevant project documents|
|**Key Output**|Final product/service/result transition, lessons learned register updates, OPA updates|

**What happens during closing:**

- Verify all work is complete and deliverables are accepted
- Transfer the final product/service/result to the next phase, production, or operations
- Collect and archive all project information
- Audit project success or failure
- Capture lessons learned and transfer knowledge
- Release organizational resources
- Close all procurement activities (contract closure)
- Perform final financial closure (confirm all payments, settle claims)
- Formally notify stakeholders that the project is complete

> ⚠️ **Don't skip closing.** Even cancelled projects need formal closure — to archive information, release resources, capture lessons learned, and provide organizational accountability.

# 🔑 Step 3 Master Summary

## Process Group Interaction Model

Here's a simplified view of how the Process Groups interact in a feedback loop:

```
        INITIATING
            │
            ▼
        PLANNING ◄──────────────┐
            │                    │
            ▼                    │ (change requests,
        EXECUTING ──────────────┤  corrective actions,
            │                    │  updated plans)
            ▼                    │
    MONITORING & ───────────────┘
    CONTROLLING
            │
            ▼
        CLOSING
```

**This is NOT purely linear.** M&C feeds back into Planning constantly. Executing generates changes that require re-planning. The cycle continues until the project is complete.

## Key Inputs/Outputs That Flow Across Groups

Understanding what flows between Process Groups is critical:

|From|To|What Flows|
|---|---|---|
|**Initiating → Planning**|Project charter, stakeholder register, assumption log||
|**Planning → Executing**|Project management plan, project documents (baselines, registers, etc.)||
|**Executing → M&C**|Work performance data, deliverables, change requests, issue log updates||
|**M&C → Planning**|Approved change requests, corrective/preventive actions||
|**M&C → Executing**|Approved changes to implement||
|**M&C → Closing**|Accepted deliverables, verified deliverables||
|**Closing → Organization**|Archived project information, lessons learned, released resources, final product||

### The Lessons Learned Register — Universal Document

The **lessons learned register** appears as an input or output in processes across virtually **all Process Groups and Knowledge Areas**. It is:

- Created as an output of Direct and Manage Project Work (4.3) in Executing
- Updated continuously throughout the project
- Used as an input to almost every subsequent process
- Finalized and archived during Close Project or Phase (6.1)

### The Most-Connected Documents

|Document|Role|
|---|---|
|**Risk register**|Input/output across Risk, Schedule, Cost, Quality, Resource, Procurement — appears in the most processes|
|**Lessons learned register**|Updated by nearly every process; used as input to nearly every process|
|**Assumption log**|Created in Initiating; updated throughout; closed in Closing|
|**Stakeholder register**|Created in Initiating; updated across Planning, Executing, M&C|
|**Issue log**|Created in Executing; updated across all groups|

## Process Frequency Summary

|Frequency|Processes|Count|
|---|---|---|
|**Once or at predefined points**|Develop Charter, Develop PM Plan, Plan Scope, Define Scope, Create WBS, Plan Schedule, Determine Budget, Plan Quality, Plan Resource, Plan Risk, Plan Procurement, Plan Cost, Close Project|~13|
|**Periodically as needed**|Collect Requirements, Estimate Costs, Estimate Activity Resources, Plan Comms, Plan Stakeholder Engagement, Acquire Resources, Conduct Procurements, Validate Scope, Identify Stakeholders|~9|
|**Continuously throughout**|Define Activities, Sequence Activities, Estimate Durations, Develop Schedule, Direct & Manage, Manage Knowledge, Manage Quality, Develop Team, Manage Team, Manage Comms, Implement Risk Responses, Manage Stakeholder Engagement, all M&C processes, all Risk identification/analysis/response processes|~27|

> 💡 **Roughly half the processes run continuously.** This reinforces that project management is an ongoing effort, not a one-time planning exercise.
